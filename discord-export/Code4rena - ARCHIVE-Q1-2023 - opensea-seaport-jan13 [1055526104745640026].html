0age 01/13/2023 22:07
Hey wardens - something we meant to include in the README but forgot to mention is that we are not interested in gas opts for the conduit controller. During our last contest we received a lot of them — we actually didn't implement any because the code is highly sensitive and not executed frequently, so readability and comprehension is the top priority. As a reminder, this same logic (no gas optimizations where they damage readability, eg assembly) applies to the reference contracts and constructor logic (and, for the love of all that is good, test contracts). Submissions against any in-scope contract are always welcome if they improve readability and comprehension without sacrificing on safety or (in the core Seaport contract) efficiency.

0age 01/20/2023 16:37
If you want to plug on-chain liquidity into Seaport you can use a contract order, where the offerer is a contract that implements the ContractOffererInterface and approves Seaport or a conduit; then, contract offerers just return orders that meet their own requirements, without requiring a signature or anything. You could imagine liquidation engines for NFT lending, pool contracts like Sudoswap, oracle-based "strategies" that will buy and sell NFTs based on the current floor price, contracts for wrapping and unwrapping ETH, etc.
